# 法律材料审核与规范化模块 — LLM 规范 v3.0

## 变更说明（v2.0 → v3.0）

v2.0 的 `NormalizedMaterial` 输出为**按九步法步骤组织的自然语言文本字段**，而 jiubufa-v2
的 `CaseInput` 要求**结构化 Pydantic 对象**（带 ID、类型枚举、数值字段等）。两者之间没有转换层。

v3.0 的核心变更：**阶段二规范化输出改为 `NormalizedCaseInput`，字段结构与 `CaseInput`
一一对应，可直接作为 `/api/workflow/run` 的 `case_input` 字段送入九步法工作流。**

---

## 模块定位

本模块是九步法审案系统的**前置材料处理网关**。完成两件事：

1. **材料充足性审核** — 双维度检查（案由专项材料清单 × 九步法输入需求），不足则指明缺失项并要求补充
2. **材料规范化** — 将用户口语化、碎片化的原始材料转化为**可直接送入 CaseInput 的结构化对象**

---

## 总架构

```
用户原始材料（口语化、碎片化）
        │
        ▼
┌─ 阶段一：材料充足性审核（双维度）─────────────────────────┐
│  维度一：案由专项材料核对                                  │
│  维度二：九步法输入需求诊断                                │
│  综合判定 → MaterialReviewResult                          │
└──────────────────────────────┬───────────────────────┘
                               │ can_proceed = true
                               ▼
┌─ 阶段二：材料规范化 → CaseInput ───────────────────────┐
│  将原始材料直接映射为 CaseInput 结构化字段               │
│  → NormalizedCaseInput（≈ CaseInput 结构，可直接送入）  │
└──────────────────────────────┬───────────────────────┘
                               │
                               ▼
              POST /api/workflow/run
              { "case_input": <NormalizedCaseInput> }
                               │
                               ▼
                     九步法审案工作流
```

# 第一部分：阶段一 — 材料充足性审核

> 阶段一设计基本沿用 v2.0，仅在 `StepRequirementCheck` 中增加了对 CaseInput
> 各子对象字段的逐项检查注释。完整的案由专项材料清单、否认vs抗辩区分、多层要件分解
> 等内容保持不变，此处不再重复。详见 `material_review_skill.md` v2.0。

阶段一输出 Schema 不变（`MaterialReviewResult` + `CaseTypeMaterialCheck` + `StepRequirementCheck`）。

阶段一新增一个字段用于对接阶段二：

```python
class MaterialReviewResult(BaseModel):
    # ... 原有字段保持不变 ...
    
    # 新增：为阶段二提供 case_module 提示
    case_module: Literal[
        "合同纠纷", "婚姻家庭", "侵权纠纷", "劳动争议",
        "民间借贷", "其他民事案件", "无法确定"
    ]
```

# 第二部分：阶段二 — 材料规范化 → CaseInput

## 2.0 核心设计原则

**正常化输出 = 可直接反序列化为 CaseInput 的结构化 JSON。**

不再产生自然语言描述（"据原告陈述，双方于2024年1月10日签订合同..."），
而是直接产出一组结构化对象列表，每个对象对应 CaseInput 的一个子组件。

### 2.0.1 字段产出能力分级

并非所有 CaseInput 字段都能从原始材料中可靠提取。按 LLM 提取能力分三级：

| 级别 | 含义 | 处理方式 |
|------|------|----------|
| **L1 — 可直接提取** | 用户原文明确提供了该信息 | LLM 直接产出，标注来源 |
| **L2 — 可推断但需标注** | 可从上下文推断但原文未明确 | LLM 产出推断值 + `[待核实]` 标记 |
| **L3 — 无法提取** | 需要法律专业知识或庭审程序才能确定 | 留空/null，由九步法工作流补充 |

### 2.0.2 字段映射总览

```
NormalizedCaseInput（阶段二输出）
│
├── case_basic_info: CaseBasicInfo        ← L1/L2
├── party_info: List[PartyInfo]           ← L1（身份信息）
├── claims: List[ClaimObject]             ← L1（诉请文本）
├── claim_facts: List[ClaimFactObject]    ← L1（事实描述 + 时间线）
├── defense_opinions: List[DefenseObject] ← L1（被告答辩 / 否认/抗辩区分）
├── counterclaims: List[CounterclaimObject] ← L1/L3
├── evidence_list: List[EvidenceObject]   ← L1（证据枚举 + 证明目的）
├── cross_examinations: List[CrossExaminationObject] ← L3（质证意见需庭审）
├── court_records: List[str]             ← L1（如有庭审记录原文）
├── legal_arguments: List[LegalArgumentObject] ← L1/L2（当事人引用法条）
├── procedural_info: Optional[ProceduralInfo]  ← L2/L3
└── existing_judgment_or_mediation: Optional[str] ← L1
```

## 2.1 case_basic_info — 案件基本信息

### 提取规则

| CaseBasicInfo 字段 | 提取来源 | 级别 | 规则 |
|---|---|---|---|
| `case_id` | 用户如提供了案号 | L2 | 保留原文案号；无则为 null |
| `case_name` | 从当事人 + 案由推断 | L2 | 格式："{原告}诉{被告}{案由}纠纷案" |
| `case_cause_text` | 从阶段一的 case_module 或原文推断 | L1/L2 | 使用标准案由名称 |
| `court` | 用户提到管辖法院 | L2 | 原文提取；无则为 null |
| `procedure_stage` | 原文描述推断 | L2 | 一审/二审/再审/执行异议/程序性审查；无则为 null |
| `filing_date` | 原文提到起诉时间 | L2 | 统一为 YYYY-MM-DD；无则为 null |
| `material_sources` | 逐项列出用户提交的材料类型 | L1 | 如 ["起诉状", "微信聊天记录", "转账凭证"] |
| `case_summary` | 从全文提取关键事实 | L1 | 2-5 句中性摘要，标注来源 |

### 规范化约束

- `case_summary` 必须中性，禁止 "显然违约""恶意拖欠" 等倾向性表述
- 所有推断值标注 `[待核实]`（通过 `case_summary` 中的来源标注体现）

---

## 2.2 party_info — 当事人信息

### 提取规则

对用户原文中出现的每一方，提取为独立的 `PartyInfo` 对象：

| PartyInfo 字段 | 提取来源 | 规则 |
|---|---|---|
| `party_id` | 自动生成 | 格式 "p{n}"，原告在前被告在后 |
| `party_name` | 原文提取 | 全名/公司全称；不完整标注 `[待核实：全名]` |
| `party_role` | 角色推断 | 原告/被告/第三人/申请人/被申请人 |
| `legal_status` | 上下文推断 | 自然人/法人/非法人组织 |
| `relationship_to_case` | 原文提取 | 如 "买卖合同的出卖人""借款合同的出借人" |
| `identity_evidence` | 原文提取 | 如用户提到 "有营业执照""身份证号XXX" |
| `standing_issue` | 推断 | 默认 false；如被告信息明显不完整设为 true |

### 格式化示例

用户输入：
> 我是做小生意的，有个客户李某从我这买了混凝土，欠我8万块钱一直不给。

输出：
```json
{
  "party_id": "p1",
  "party_name": "[待核实：原告全名]",
  "party_role": "原告",
  "legal_status": "自然人",
  "relationship_to_case": "买卖合同的出卖人",
  "identity_evidence": [],
  "standing_issue": false
},
{
  "party_id": "p2",
  "party_name": "李某",
  "party_role": "被告",
  "legal_status": "自然人",
  "relationship_to_case": "买卖合同的买受人",
  "identity_evidence": [],
  "standing_issue": true
}
```

注意：被告身份信息不完整 → `standing_issue = true`，这会在阶段一的 `can_proceed` 中触发警告。

## 2.3 claims — 诉讼请求

### 提取规则

从用户原文中逐条提取诉讼请求，每条生成一个 `ClaimObject`：

| ClaimObject 字段 | 提取规则 | 级别 |
|---|---|---|
| `claim_id` | 自动生成 "c{n}" | — |
| `claim_text_original` | **逐字保留**用户原文诉请 | L1 |
| `claim_text_normalized` | 法言法语化版本 | L1 |
| `claimant` | 原告 party_id | L1 |
| `respondent` | 被告 party_id | L1 |
| `claim_type_candidate` | 从诉请类型推断 | L2 |
| `object_type` | 金钱/物/行为/状态确认 | L2 |
| `amount` | 从原文提取数值（float） | L1 |
| `behavior_requested` | 要求对方做什么 | L1 |
| `is_clear` | 诉请是否明确无歧义 | L2 |
| `is_executable` | 是否便于实际执行 | L2 |
| `conflict_with_other_claims` | 多条诉请间是否有矛盾 | L3（默认 []） |
| `supplement_needed` | 是否需释明补充 | L2 |
| `priority_type` | primary/alternative/parallel/selective | L2 |

### claim_type_candidate 枚举

```
合同履行请求权 / 金钱给付请求权 / 确认请求权 / 形成请求权 /
损害赔偿请求权 / 违约责任 / 侵权责任 / 不当得利返还请求权 /
物权保护请求权 / 身份关系确认 / 抚养费给付 / 离婚 /
...[其他，尽量匹配现有枚举]
```

### 金额提取规则

- 原文 "8万" → `amount: 80000.0`
- 原文 "五千元" → `amount: 5000.0`
- 原文 "按 LPR 计算利息" → `amount: null`（金额不固定）
- 金额不明确 → `amount: null`，`supplement_needed: true`

## 2.4 claim_facts — 事实主张

### 提取规则

将用户描述的事实按时间线排列，每个独立事实生成一个 `ClaimFactObject`：

| ClaimFactObject 字段 | 提取规则 | 级别 |
|---|---|---|
| `fact_id` | 自动生成 "f{n}" | — |
| `fact_text_original` | **逐字保留**用户原文该段事实 | L1 |
| `fact_text_normalized` | 法言法语化 + 来源标注 | L1 |
| `fact_time` | 从原文提取时间，统一 YYYY-MM-DD | L1 |
| `fact_actor` | 实施该行为的主体 party_id | L1 |
| `fact_counterparty` | 该行为涉及的对造 party_id | L1 |
| `fact_type_candidate` | 合同成立/履行行为/违约行为/侵权行为/... | L2 |
| `linked_claim_id` | 该事实服务于哪个诉请 | L2 |
| `linked_evidence_ids` | 该事实由哪些证据支持 | L2 |
| `possible_fact_slot` | 该事实对应哪个要件槽位 | L3（默认 []） |
| `clarity_status` | 明确/模糊/矛盾/遗漏 | L2 |
| `opponent_response` | 对方是否承认/否认/抗辩 | L2 |

### 来源标注约束（五大硬性约束第1条）

规范化后的 `fact_text_normalized` 必须在每句末尾标注信息来源：
- "（据原告陈述）"
- "（据微信聊天记录）"
- "（据转账凭证）"
- "（据被告自认）"

### 时间线排序

事实按 `fact_time` 升序排列。模糊时间（"去年""不久前"）放在时间线末尾，
标注 `[时间待核实]`。

## 2.5 defense_opinions — 答辩意见

### 提取规则

从被告的答辩/回应中提取，生成 `DefenseObject` 列表：

| DefenseObject 字段 | 提取规则 | 级别 |
|---|---|---|
| `defense_id` | 自动生成 "d{n}" | — |
| `defense_text_original` | **逐字保留**被告原文答辩 | L1 |
| `defense_text_normalized` | 法言法语化版本 | L1 |
| `defense_target_claim_id` | 该答辩针对哪个诉请 | L2 |
| `response_type` | **关键区分**：承认 / 否认 / 抗辩 / 抗辩权 / 程序性异议 | L1/L2 |
| `defense_type_candidate` | 同时履行抗辩权/质量瑕疵抗辩/时效抗辩/... | L2 |
| `new_fact_asserted` | 是否提出了新的事实主张 | L1 |
| `linked_evidence_ids` | 答辩所依据的证据 | L2 |
| `possible_defense_basis` | 可能的法条依据 | L3（默认 []） |
| `clarification_needed` | 是否需释明 | L2 |

### response_type 判定规则（最关键的区分）

```
被告说 "我没借过钱"           → 否认（denial）
被告说 "借过但已经还了"       → 抗辩（defense）— 权利消灭抗辩
被告说 "借过但已过诉讼时效"   → 抗辩权（defense_right）— 权利阻止抗辩
被告说 "法院没管辖权"         → 程序性异议（procedural）
被告说 "对，欠这个钱"         → 承认（admission）
被告说 "我会付的"             → 自认（admission）— 民事证据规定第3条
```

**举证责任影响**：
- 否认 → 不转移举证责任
- 抗辩 → 被告对抗辩事实承担举证责任
- 抗辩权 → 被告对抗辩权成立要件承担举证责任

## 2.6 evidence_list — 证据清单

### 字段映射：EvidenceItem(规范) → EvidenceObject(jiubufa-v2)

| EvidenceObject 字段 | 从原始材料提取 | 级别 |
|---|---|---|
| `evidence_id` | 自动生成 "e{n}" | — |
| `evidence_name` | 原文提取（"微信聊天记录""借条""转账记录"） | L1 |
| `submitted_by` | 推断（p1/p2） | L1 |
| `evidence_type` | 书证/物证/电子数据/证人证言/视听资料/鉴定意见/勘验笔录/当事人陈述 | L1 |
| `proof_purpose_original` | 原文提取证明目的 | L1 |
| `proof_purpose_normalized` | 法言法语化证明目的 | L1 |
| `linked_claim_id` | 支持哪个诉请 | L2（可推断则填，否则 null） |
| `linked_defense_id` | 支持哪个答辩 | L2 |
| `linked_fact_ids` | 支持哪些事实 | L2 |
| `linked_element_ids` | 支持哪些要件 | L3（默认 []） |
| `opponent_cross_examination` | 对方质证意见 | L3（默认 null） |
| `legality_status` | 合法性判断 | L3（默认 null） |
| `relevance_status` | 关联性判断 | L3（默认 null） |
| `authenticity_status` | 真实性判断 | L3（默认 null） |
| `probative_force` | 证明力 | L3（默认 null） |
| `adopted_status` | 采信状态 | L3（默认 null） |

### v3.0 新增字段（jiubufa-v2 缺少但规范要求）

在 `EvidenceObject` 基础上，阶段二额外产出以下**审核用元数据**（不进入 CaseInput，
但随 `NormalizedCaseInput` 一起返回供前端展示）：

```python
class EvidenceMeta(BaseModel):
    evidence_id: str            # 对应 EvidenceObject.evidence_id
    subtype: str = ""           # 电子数据子类：微信聊天记录/银行转账电子回单/电子邮件/...
    completeness: str = "待核实" # 完整/部分/待核实
    authenticity_note: str = "" # 原件/复印件/原始载体/待核验
    standalone_capable: bool = True  # 《民事证据规定》第90条
    standalone_limitation: str = ""  # 如不能单独定案，说明原因
```

**证据类型 → subtype 映射**：

| evidence_type | 常见 subtype |
|---|---|
| 电子数据 | 微信聊天记录 / 银行转账电子回单 / 电子邮件 / 手机短信 / 支付宝转账记录 / 网页截图 |
| 书证 | 合同原件 / 借条 / 欠条 / 送货单 / 签收单 / 发票 / 催款函 |
| 视听资料 | 录音 / 录像 / 照片 |
| 鉴定意见 | 司法鉴定报告 / 第三方检测报告 / 价格评估报告 |

### 《民事证据规定》第90条检查

以下证据**不能单独作为认定案件事实的依据**，在 `standalone_capable` 中标记：

1. 当事人的陈述
2. 无/限制民事行为能力人与其年龄智力不相符的证言
3. 与一方当事人有利害关系的证人证言
4. 存有疑点的视听资料、电子数据
5. 无法与原件原物核对的复制件、复制品

## 2.7 其他字段

### counterclaims（反诉）

```python
class CounterclaimObject:
    counterclaim_id: str       # 自动生成 "cc{n}"
```

如果用户引用了法条或提出了法律论点：

```python
class LegalArgumentObject:

    submitted_by: Optional[str]  # p1/p2

```

### procedural_info（程序事项）

```python
class ProceduralInfo:
    jurisdiction: Optional[str] = None          # 管辖法院
    limitation_period_status: Optional[str] = None  # 时效状态
    proof_period_status: Optional[str] = None       # 举证期状态
    other_notes: Optional[str] = None
```

大多为 L3 级别（需要法律专业判断），默认全部为 null。

### existing_judgment_or_mediation

如果用户提到已有判决/调解书，提取原文。无则 null。

## 2.8 规范化五大硬性约束（v3.0 适配）

1. **来源标注约束**：每个事实描述必须标注信息来源。体现在 `fact_text_normalized`、
   `defense_text_normalized`、`proof_purpose_normalized` 等字段中。
2. **缺口标注约束**：信息不明确处用 `[待核实：XXX]` 标注。体现在所有 `_normalized`
   字段和 `party_name`、`case_summary` 中。
3. **不增不删约束**：`*_original` 字段逐字保留用户原文；不添加原文中没有的事实/证据/法条。
4. **中立性约束**：不使用倾向性语言。`case_summary`、`*_normalized` 须中性。
5. **可追溯约束**：`original_input` 完整保留在返回对象中（通过 `NormalizedCaseInput`
   新增字段 `original_input: str`）。

## 2.9 阶段二 System Prompt

```
你是一名资深法院书记员，负责将当事人提交的口语化、碎片化材料整理为
结构化的案件数据，直接输入"要件审判九步法"AI审案系统。

你的输出必须是严格的 JSON，结构完全对齐目标 Schema。不要输出描述性文本。
只输出一个 JSON 对象。

## 你的任务

从用户提交的原始材料中提取以下结构化信息，填入对应字段：

1. **案件基本信息** (case_basic_info)
   - 从全文推断案由(case_cause_text)、案件名称(case_name)
   - 列出用户提交的材料来源清单(material_sources)
   - 写2-5句中性的案件摘要(case_summary)

2. **当事人信息** (party_info)
   - 识别所有当事人，标注角色（原告/被告/第三人）
   - 提取姓名/名称、身份类型（自然人/法人）
   - 如被告信息不完整，standing_issue 设为 true

3. **诉讼请求** (claims)
   - 逐条提取，逐字保留原文(claim_text_original)
   - 归一化表述(claim_text_normalized)
   - 提取金额（数字）、请求类型(object_type、claim_type_candidate)
   - 判断诉请是否明确(is_clear)、是否可执行(is_executable)

4. **事实主张** (claim_facts)
   - 逐条提取原文事实，标注时间（统一YYYY-MM-DD）
   - 归一化表述并标注来源——"（据原告陈述）""（据微信记录）"等
   - 判断每项事实的清晰度(clarity_status)
   - 标注对方回应（承认/否认/抗辩/不明确）
   - 按时间顺序排列

5. **答辩意见** (defense_opinions)
   - 提取被告的答辩原文
   - 【关键】区分 response_type：
     否认 = 单纯否定("我没借过")→不转移举证责任
     抗辩 = 提出新事实("借过但已还")→被告举证
     抗辩权 = 主张权利障碍("已过时效")→被告举证
     承认 = 认可对方主张
   - 判断是否提出了新事实(new_fact_asserted)

6. **证据清单** (evidence_list)
   - 逐项列出所有提到的证据
   - 标注证据类型(evidence_type)、子类(subtype——由元数据返回)
   - 写明证明目的
   - 标注提供方(submitted_by)
   - 【关键】检查是否落入《民事证据规定》第90条"不能单独定案"五类情形
   - 标注原件/复印件/待核验状态

7. **其他**
   - 反诉(counterclaims)——如有
   - 庭审记录(court_records)——如有
   - 法律意见(legal_arguments)——如用户引用法条
   - 程序事项(procedural_info)——管辖/时效（如能从原文推断）
   - 已有判决/调解(existing_judgment_or_mediation)——如用户提及

## 规范化规则

### 金额提取
- "8万" → 80000.0
- "五千元" → 5000.0
- 不明确的金额 → null

### 时间统一
- YYYY-MM-DD 格式
- "去年9月" → 推断年份 + "-09" + [时间待核实]
- 完全模糊 → null

### 禁止事项
- 禁止编造法条编号
- 禁止编造证据内容
- 禁止推测未陈述的事实
- 禁止删除用户原文中的任何事实
- 禁止使用"显然""恶意"等倾向性词汇
- 禁止在 *_original 字段中修改用户原文
```
