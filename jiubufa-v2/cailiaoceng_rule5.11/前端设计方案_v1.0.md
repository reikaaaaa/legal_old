# 九步法审案系统 — 前端设计方案 v1.0

## 基于现有后端能力的页面、数据流与接口衔接设计

---

# 第一部分：当前状态分析

## 1.1 已有前端现状

`front_test/jiubufa.html`（89KB 单页应用）采用庄严肃穆的古典司法美学设计，结构为三标签页：

| 标签 | 名称 | 当前功能 |
|------|------|----------|
| 壹 | 案件录入 | 结构化表单（当事人、诉请、事实、证据） + JSON 导入/导出 + 快速评分 |
| 貳 | 工作流执行 | 步骤阶梯动画 + 计时器 + 进度模拟 |
| 叁 | 裁判分析 | 综合评分仪表盘 + 裁决印章 + 裁判文书框架 + 一致性校验 |

**当前设计局限：**
- 材料审核/规范化流程完全缺失
- 只有结构化表单输入，无口语化文本输入入口
- 无补正提示交互
- 九步法输入直接使用表单拼接的 JSON，未经材料规范层处理
- 结果展示为一次性全量渲染，无折叠/展开控制

## 1.2 已有后端能力清单

### 材料规范层（cailiaoceng_rule5.11）

| 能力 | 状态 | 对应数据结构 |
|------|------|-------------|
| 材料审核（阶段一） | ✅ 已实现 | `MaterialReviewResult` |
| 材料规范化（阶段二） | ✅ 已实现 | `NormalizedCaseInput` |
| 串联流水线 | ✅ 已实现 | `MaterialFullResult` |
| 适配器 | ✅ 已实现 | `adapter.to_case_input()` |

### 九步法引擎（jiubufa-v2）

| 能力 | 状态 | API 端点 |
|------|------|----------|
| 完整工作流 | ✅ | `POST /api/workflow/run` |
| 仅评分 | ✅ | `POST /api/workflow/score_only` |
| 健康检查 | ✅ | `GET /api/health` |
| 知识库统计 | ✅ | `GET /api/kb/stats` |

# 第二部分：整体页面结构设计

## 2.1 用户流程主线

```
┌─────────────────────────────────────────────────────────────┐
│ 阶段A: 材料提交                                              │
│                                                             │
│  用户输入原始案件材料（文本框 + 身份选择 + 案由选择）           │
│         │                                                   │
│         ▼                                                   │
│  材料规范层处理（审核 + 规范化）                               │
│         │                                                   │
│    ┌────┴────┐                                              │
│    │ 审核通过  │  审核未通过                                  │
│    ▼         ▼                                              │
│  进入阶段B   弹出补正弹窗 → 补充材料 → 重新审核                │
│             或"暂不补充，继续" → 进入阶段B（带风险提示）        │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ 阶段B: 九步法确认与提交                                       │
│                                                             │
│  展示规范化后的材料（CaseInput 预览）                          │
│  用户可修改、确认                                              │
│  选择模型 → 提交九步法                                        │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ 阶段C: 结果展示                                               │
│                                                             │
│  总结论优先 → 步骤卡片可展开 → 法条侧框 → 证据专题             │
│  支持下载 / 修改 / 二次分析                                    │
└─────────────────────────────────────────────────────────────┘
```

## 2.2 页面区域划分

建议将当前三标签页扩展为以下页面/区域结构：

```
┌─────────────────────────────────────────────────────┐
│ 顶部栏：标题 + 系统状态指示器 + 运行计时器             │
├──────────────┬──────────────────┬───────────────────┤
│              │                  │                   │
│  材料提交区   │  九步法确认与    │  结果展示区        │
│  (阶段A)     │  执行区(阶段B)   │  (阶段C)          │
│              │                  │                   │
│  ·文本输入   │  ·规范化材料预览 │  ·总结论          │
│  ·身份/案由  │  ·模型选择下拉   │  ·评分仪表盘      │
│  ·提交按钮   │  ·提交按钮       │  ·步骤卡片组      │
│  ·审核结果   │  ·状态提示       │  ·裁判文书框架    │
│  ·补正弹窗   │                  │  ·法条侧框        │
│              │                  │  ·证据分析        │
│              │                  │  ·错误/警告区     │
│              │                  │  ·系统详情(折叠)   │
└──────────────┴──────────────────┴───────────────────┘
```

**设计原则：**
- 三阶段为**渐进式流程**，非三个独立标签页
- 阶段A → 阶段B 为自动衔接（审核通过时）或手动衔接（用户选择"暂不补充"）
- 阶段B → 阶段C 为提交后自动跳转
- 阶段C 允许返回阶段B 修改后重跑

# 第三部分：阶段A 详细设计 — 材料提交与审核

## 3.1 材料输入区

| 组件 | 字段来源 | 说明 |
|------|----------|------|
| 文本输入框 A | `str → MaterialPipeline.review(raw)` | 大文本框，支持多行输入，placeholder 示例引导 |
| 用户身份选择 | 前端枚举 | 原告/被告/第三人/代理人/其他 |
| 案件类型选择 | 前端枚举 + `case_module` 回填 | 系统自动识别/合同纠纷/婚姻家庭/侵权纠纷/劳动争议/民间借贷/其他 |
| 提交按钮 1 | 触发 `POST /material/review`（待实现 API 化） | 按钮文案："提交审核" |

**交互说明：**
- 用户可仅填写文本框A 即提交，身份和案由为辅助输入
- 案由默认为"系统自动识别"，此时由 LLM 自动判定；用户也可手动指定
- 提交后进入材料规范层处理，按钮进入 loading 状态

## 3.2 审核结果展示区

**审核通过时（can_proceed = true）：**

| 展示项 | 数据来源 | 展示方式 |
|--------|----------|----------|
| 识别案由 | `MaterialReviewResult.case_module` | 标签/徽标 |
| 综合评定 | `MaterialReviewResult.overall_status` | 状态文本（如"材料基本完整"）+ 绿色通过图标 |
| 核心材料提供率 | `CaseTypeMaterialCheck.core_provided_rate` | 环形进度条 + 百分比 |
| 置信度 | `MaterialReviewResult.confidence` | 标签（高=绿/中=黄/低=红） |
| 逐项材料检查 | `CaseTypeMaterialCheck.checklist[]` | 折叠表格，每行展示 name/status/description |
| 九步法逐步诊断 | `MaterialReviewResult.step_checks[]` | 折叠卡片组 |
| 规范化材料预览 | `NormalizedCaseInput`（阶段二产出） | 见 3.3 |

**审核未通过时（can_proceed = false）：**

| 额外展示 | 数据来源 |
|----------|----------|
| 缺失核心材料清单 | `MaterialReviewResult.missing_core_materials[]` |
| 缺失可选材料清单 | `MaterialReviewResult.missing_optional_materials[]` |
| 补正指引 | `MaterialReviewResult.upload_instructions` |
| 补正弹窗触发 | **自动弹出**（见 3.3） |

## 3.3 补正提示弹窗

**触发条件：** `can_proceed == false`

| 弹窗区域 | 字段来源 | 说明 |
|----------|----------|------|
| 缺失材料名称 | `missing_core_materials[]` + `missing_optional_materials[]` | 分类展示（核心/可选），每项前面带图标 |
| 材料作用说明 | 规范文档中的"缺失后果"字段 + LLM 生成 | 用通俗语言说明为什么需要这份材料 |
| 补充理由/法律依据 | `MaterialItem.legal_basis` + `special_rules_note` | 如"《民诉法》第122条要求被告明确" |
| 文本输入框 | 前端组件 | 用户在此输入补充材料内容 |
| 提交按钮 2 | 触发重新审核 | 文案："补充并重新审核" |
| 关闭按钮 | 前端交互 | 文案："暂不补充，继续分析" |

**"暂不补充"交互逻辑（重要）：**

```
用户点击"暂不补充，继续分析"
  │
  ├── 前端记录：用户已知晓材料缺失
  ├── 前端保留当前审核结果（不丢弃）
  ├── 弹窗关闭
  ├── 阶段A 审核结果区显示"材料存在缺失"警告标记
  ├── 阶段B 可用（规范化材料仍基于现有内容生成）
  └── 阶段C 总结论区域展示"分析结果可能受材料不足影响"
```

**材料补充后重新提交逻辑：**

```
用户输入补充材料 → 点击"补充并重新审核"
  │
  ├── 前端将 文本输入框A 内容 + 补充文本 合并
  ├── 调用 材料审核 API（重新执行阶段一）
  ├── 审核结果刷新展示
  ├── 如果 can_proceed=false → 弹窗再次弹出（更新缺失清单）
  └── 如果 can_proceed=true → 弹窗关闭，进入阶段B
```

## 3.4 规范化材料展示区（阶段B 入口）

**无论审核是否通过**（用户选择"暂不补充"后也进入此区域），前端展示规范化后的材料内容。

| 展示项 | 数据来源 | 展示方式 |
|--------|----------|----------|
| 当事人信息 | `NormalizedCaseInput.party_info[]` | 卡片列表 |
| 诉讼请求 | `NormalizedCaseInput.claims[]` | 编号列表，含金额/类型 |
| 事实主张 | `NormalizedCaseInput.claim_facts[]` | 时间线展示 |
| 答辩意见 | `NormalizedCaseInput.defense_opinions[]` | 卡片，区分否认/抗辩 |
| 证据清单 | `NormalizedCaseInput.evidence_list[]` | 表格 + 证据元数据 |
| 案件摘要 | `NormalizedCaseInput.case_basic_info.case_summary` | 文本段落 |

**关键按钮：** "一键提交至九步法" — 将 `NormalizedCaseInput` 转换为 `CaseInput` 后送入阶段B

**⚠️ 待确认：** 当前材料规范层未暴露为 HTTP API。前端需直接调用 `MaterialPipeline.full()` 或等待后端提供 `/material/full` 端点。当前设计假设存在一个统一的 `/material/full` POST 接口，接收 `{raw_material: str}` 并返回 `MaterialFullResult`。

# 第四部分：阶段B 详细设计 — 九步法确认与提交

## 4.1 九步法输入预览区

此区域不是空白文本框，而是**结构化预览 + 编辑区**。

| 子区域 | 数据来源 | 展示与交互 |
|--------|----------|------------|
| 案件基本信息 | `case_basic_info` | 只读展示（case_name, court, procedure_stage） |
| 当事人信息 | `party_info[]` | 卡片展示，支持编辑 |
| 诉讼请求 | `claims[]` | 表格展示（编号/内容/金额/类型/是否明确），支持编辑 |
| 事实主张 | `claim_facts[]` | 时间线展示，支持编辑和排序 |
| 答辩意见 | `defense_opinions[]` | 卡片展示，标注否认/抗辩类型 |
| 证据列表 | `evidence_list[]` | 表格展示（名称/类型/来源/证明目的/完整性），支持展开 evidence_meta |
| 来源追溯 | `original_input` | 折叠区域，展示原始输入文本 |

**设计要点：**
- 预览区默认**只读**，提供"编辑"切换按钮
- 用户可以修改 LLM 提取有误的字段（如金额识别错误）
- 修改后不重新调用材料规范层，而是直接更新前端状态
- 提交九步法时将编辑后的内容作为 `CaseInput` 发送

## 4.2 模型选择与提交区

| 组件 | 数据来源 | 行为 |
|------|----------|------|
| 模型下拉框 | `GET /api/health → models` | 展示可用模型列表（dashscope-qwen-plus 等） |
| 提交按钮 | 触发 `POST /api/workflow/run` | 文案："开始九步法分析"，提交后变为"分析中…" |
| 状态提示 | `can_proceed` + `fallback_user_choice` | 显示当前材料完整度状态 |

**⚠️ 待确认：** 当前 `POST /api/workflow/run` 的 `RunWorkflowRequest.model_name` 字段需要后端实际支持模型切换。`api.py` 第124-126行仅做了模型名校验但未实际注入到 LLMClient。建议在前端设计中保留该下拉框，但在后端支持前默认置灰或标注为"当前使用默认模型"。

## 4.3 提交状态流转

```
用户点击"开始九步法分析"
  │
  ├── 前端 POST /api/workflow/run { case_input: ... }
  ├── 进入 loading 状态（步骤进度指示器）
  │
  ├── status = "ok"        → 跳转阶段C（展示结果）
  ├── status = "blocked"   → 展示阻断原因 + 建议补充
  └── status = "awaiting_user_choice" → 展示用户选择门
```

**awaiting_user_choice 处理：**

```
展示 FallbackGate.available_choices
  ├── "补充材料" → 回退到阶段A补正弹窗
  ├── "继续弱裁判" → 重新 POST /api/workflow/run（带上 fallback_user_choice）
  ├── "仅查看要件和争点" → 同上
  └── "附风险提示继续" → 同上 (proceed_with_risk_notes)
```

# 第五部分：阶段C 详细设计 — 结果展示

## 5.1 总结论优先区（页面顶部）

| 展示项 | 数据来源 |
|--------|----------|
| 案件综合结论 | `strong_judgment.document_skeleton.判决主文` 或 `weak_judgment.weak_subsumption_results[].tentative_judgment_result` |
| 结果模式标签 | `strong_judgment.mode` / `weak_judgment.mode` / `partial_output.mode` |
| 风险等级 | `strong_judgment.risk_level` / `weak_judgment.risk_level` |
| 材料充足度评分 | `strong_judgment.sufficiency_score` / `weak_judgment.sufficiency_score`（环形仪表盘） |
| 材料不足警告 | `weak_judgment.missing_inputs[]` / `weak_judgment.unsupported_elements[]` |
| 补充建议 | `weak_judgment.upgrade_to_strong_judgment_requirements[]` |
| 核心争议点 | `step6.issues[]`（筛选 priority=high） |

## 5.2 九步法步骤展示区

**展示策略：折叠卡片组**

默认状态：每步仅展示摘要行（步骤序号 + 名称 + 状态图标 + 摘要文字）

| 步骤默认展示 | 数据来源 |
|-------------|----------|
| 步骤名称 | 硬编码映射（Step1=固定权利请求…） |
| 状态图标 | 见下方映射表 |
| 单行摘要 | 见下方映射表 |

**状态图标映射：**

| 来源场景 | 图标 | 颜色 |
|----------|------|------|
| `step{1-9} != None` | ✓ | 绿 |
| `step{1-9} == None` 且 `errors` 中无对应错误 | ○（未执行） | 灰 |
| `step{1-9} == None` 且 `errors` 中有对应错误 | ✗ | 红/朱砂 |

**展开后各步骤详情：**

| 步骤 | 摘要来源 | 详情字段 | 关联法条来源 |
|------|----------|----------|-------------|
| Step1 固定权利请求 | `fixed_claims` 数量和类型 | `fixed_claims[].claim_text_normalized`, `issues`, `clarification_questions` | — |
| Step2 请求权基础 | `request_basis_candidates` 数量 | `request_basis_candidates[].rule_unit_ref`, `selection_reason`, `risk_note` | `rule_unit_ref` 对应的法条 |
| Step3 抗辩权基础 | `defense_basis_candidates` 数量 + response_type 分布 | `defense_basis_candidates[].response_type`, `defense_type`, `risk_note` | `rule_unit_ref` 对应的法条 |
| Step4 要件分析 | `element_matrix` 行数 | `element_matrix[].element_name`, `element_type`, `burden_party`, `used_for` | `rule_unit_id` |
| Step5 诉讼主张检索 | `claim_fact_mapping` 完整/缺失/冲突统计 | `claim_fact_mapping[].assertion_status`, `clarification_question`, `risk_note` | — |
| Step6 争点整理 | `issues` 数量和类型 | `issues[].issue_text`, `issue_type`, `priority`, `linked_element_ids` | — |
| Step7 要件事实证明 | `proof_plan` 行数 | `proof_plan[].fact_to_prove`, `burden_party`, `proof_gap`, `effect_if_unknown` | — |
| Step8 事实认定 | `fact_findings` 认定/未认定/真伪不明统计 | `fact_findings[].finding_status`, `reasoning`, `adopted_evidence_ids` | — |
| Step9 要件归入 | `subsumption_results` 支持/部分/驳回统计 | `subsumption_results[].judgment_result`, `reasoning_summary`, `cited_rules` | `cited_rules[]` |

## 5.3 相关法条侧框

| 展示项 | 数据来源 | 关联信息 |
|--------|----------|----------|
| 法条名称 | `kb/loader.py` 的知识库条目（如已加载） | 法条ID |
| 条文内容 | `RuleUnitRef` → 知识库索引（待确认是否返回全文） | — |
| 适用理由 | `RequestBasisCandidate.selection_reason` / `DefenseBasisCandidate.selection_reason` | 关联步骤 |
| 关联步骤 | 该法条在哪一步被检索/引用 | Step2/3/9 |
| 关联争点/事实 | `linked_claim_id` / `linked_defense_id` | Step6/8 |

**⚠️ 待确认：** 当前 `POST /api/workflow/run` 返回的法条信息为 `rule_unit_id`（字符串ID），不包含法条全文。前端需要(b)额外请求知识库查询接口获取原文，或(b)建议后端在输出中附带法条摘要。当前建议在侧框中展示法条ID，并标注"待后端补充全文检索接口"。

# 第九部分：设计约束与不破坏原则

1. **不修改 `schemas/` 中的 Pydantic 模型** — 如有前端需要的额外字段，通过 API 层增加包装或在返回体中追加，不改变现有模型定义
2. **不修改 `orchestrator.py` 的流程分支逻辑** — 前端仅消费其输出，不干预编排决策
3. **不修改 `api.py` 的现有端点签名** — 新增端点独立添加，不改变已有路由
4. **不删除 `jiubufa.html` 的现有功能** — 新前端设计为渐进增强，可先以"高级模式/向导模式"与现有表单并存
5. **材料规范层保持独立模块** — 其状态为可独立调用，不与九步法引擎耦合
6. **前端状态管理保持单一数据源** — 各阶段共享同一个 `CaseInput` 对象引用，避免多份数据不一致
