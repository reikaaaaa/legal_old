# 九步法 AI 审案工作流 - 项目运行逻辑说明文档

## 一、项目概述

本项目是基于"要件审判九步法"和五层法律标签库（L1-L5）的 AI 智能审案后端系统。系统通过大语言模型（LLM）结合结构化法律知识库，自动完成从诉讼请求固定到裁判输出的完整审判推理流程。

### 核心特性

- **九步法推理**：严格按照要件审判九步法的逻辑顺序执行案件分析
- **五层标签检索**：基于 L1-L5 法律标签体系进行精准法条检索
- **联网搜索增强**：在关键步骤启用联网搜索，补充最新法律法规
- **保底裁判机制**：根据输入材料充足度评分，动态选择强裁判/弱裁判/部分输出
- **上下文传递**：每个步骤都包含完整的原始案件信息和前面步骤的输出结果
- **双入口支持**：提供命令行（CLI）和 HTTP API（FastAPI）两种调用方式
- **容错与重试**：LLM 调用失败时自动重试，步骤异常不阻断整体流程

## 三、数据流转

### 3.1 输入数据结构

**顶层输入：`CaseInput`**

```
CaseInput
├── case_basic_info          # 案件基本信息（案号、案由、法院等）
├── party_info               # 当事人信息列表
├── claims                   # 诉讼请求列表
├── claim_facts              # 事实主张列表
├── defense_opinions         # 答辩/抗辩意见列表
├── counterclaims            # 反诉列表（可选）
├── evidence_list            # 证据列表
├── cross_examinations       # 质证意见列表
├── court_records            # 庭审笔录（可选）
├── legal_arguments          # 法律意见（可选）
├── procedural_info          # 程序事项（可选）
└── fallback_user_choice     # 用户保底选择（supplement / continue_weak_judgment / partial_output_only）
```

### 3.2 中间状态：`WorkflowState`

九步法执行过程中，所有中间产物累积在 `WorkflowState` 中：

```
WorkflowState
├── case_input               # 原始输入
├── step1                    # Step1Output：固定后的诉讼请求
├── step2                    # Step2Output：请求权基础候选
├── step3                    # Step3Output：抗辩权基础候选
├── step4                    # Step4Output：构成要件矩阵
├── step5                    # Step5Output：待证事实映射
├── step6                    # Step6Output：争点列表
├── step7                    # Step7Output：举证计划
├── step8                    # Step8Output：事实认定结果
├── step9                    # Step9Output：要件归入结果
├── sufficiency_score        # 输入充足度评分
├── fallback_gate            # 保底选择门
├── timings_ms               # 各步骤耗时记录
├── errors                   # 错误列表
└── warnings                 # 警告列表
```

### 3.3 输出数据结构

**最终输出：`WorkflowResult`**

```
WorkflowResult
├── case_id                  # 案件ID
├── status                   # 状态：ok / blocked / awaiting_user_choice
├── fallback_gate            # 保底选择门（含评分和可用选项）
├── step1 ~ step9            # 各步骤中间产物
├── strong_judgment          # 强裁判输出（评分≥80）
├── weak_judgment            # 弱裁判输出（评分40-79且用户选择继续）
├── partial_output           # 部分输出（用户选择仅查看要件/争点）
├── timings_ms               # 各步骤耗时
├── errors                   # 错误列表
└── warnings                 # 警告列表
```

### 4.2 上下文传递机制

**每个步骤都接收完整的上下文信息**，通过 `context_builder.py` 构建：

```python
full_context = build_full_context(
    state,
    current_step=STEP_KEY,
    include_original_case=True,    # 包含原始案件信息
    include_previous_steps=True,   # 包含前面步骤的输出
)
```

**上下文包含两部分**：

1. **原始案件信息**（`_build_case_context`）：
   - 案件基本信息（案号、案由、法院等）
   - 诉讼请求
   - 事实主张（截断避免过长，默认3000字符）
   - 答辩意见
   - 证据清单（简化版）
   - 庭审笔录摘要

2. **前面步骤的输出结果**（`_build_steps_context`）：
   - 自动识别当前步骤在九步法中的位置
   - 提取所有前面步骤的关键输出
   - 格式化为可读文本（如 Step1的固定诉请、Step2的请求权基础等）

**优势**：
- LLM 在每个步骤都能看到完整的案件全貌
- 避免信息丢失，保证推理连贯性
- 支持跨步骤的信息引用和校验

### 4.3 九步法详细流程

#### **Step 1：固定权利请求**

**输入**：原始案件输入（claims、party_info、claim_facts 等）

**处理**：
- 规范化每项诉讼请求（claim_text_normalized）
- 判断请求是否明确（is_clear）、可执行（is_executable）
- 映射到 L3 claim_type 标签（如 payment_claim、damages_claim）
- 推断案件 legal_domain 和 case_cause
- 识别请求权竞合/聚合/备位关系
- 对不明确请求生成释明问题

**输出**：`Step1Output`（fixed_claims、case_cause_inferred、legal_domain_inferred）

**上下文**：仅包含原始案件信息

---

#### **Step 2：确定请求权基础规范**

**输入**：Step 1 的 fixed_claims + 知识库检索结果 + 联网搜索法条

**处理**：
1. **本地检索**：用 L2/L3 标签从知识库检索候选规则单元（top-k=12）
2. **联网搜索**（如果启用）：使用 DeepSeek V4 Flash 搜索最新法律法规（top-k=2）
3. **LLM 选择**：为每项请求选择最适合的请求权基础规范
4. 遵循：具体规则 > 原则规则；特别规则 > 一般规则
5. 处理请求权竞合，标注 priority（primary/alternative/supplementary）
6. 说明选择理由（selection_reason）

**输出**：`Step2Output`（request_basis_candidates、competition_analysis）

**上下文**：原始案件信息 + Step1 输出

**联网搜索**：
- 为每个诉请（最多3个）搜索相关法条
- 查询条件：claim_text + case_cause + legal_domain + claim_type
- 返回法条包含：law_name、article_no、article_text、effective_status、relevance_reason

---

#### **Step 3：确定抗辩权基础规范**

**输入**：defense_opinions + fixed_claims + request_basis + 知识库检索 + 联网搜索法条

**处理**：
1. **本地检索**：用 L3 defense_type 标签检索候选规则单元（top-k=10）
2. **联网搜索**（如果启用）：搜索抗辩相关法条（top-k=2）
3. 将答辩分类为：承认/否认/抗辩/抗辩权/程序性异议
4. 区分"否认"（非真正抗辩）与"真正抗辩"（需抗辩基础）
5. 映射到 L3 defense_type 标签
6. 为真正抗辩检索并选择支持规则单元
7. 标注需要释明的抗辩

**输出**：`Step3Output`（defense_basis_candidates）

**上下文**：原始案件信息 + Step1、Step2 输出

**联网搜索**：
- 为每个答辩意见（最多3个）搜索相关法条
- 查询条件：defense_text + case_cause + legal_domain + defense_type

---

#### **Step 4：构成要件分析**

**输入**：request_basis + defense_basis + 知识库规则单元的 L4 要件

**处理**：
- 从选定的请求权/抗辩权基础中提取构成要件
- 构建要件矩阵（element_matrix）
- 标注每个要件的：
  - 要件类型、逻辑关系（AND/OR/NOT）
  - 隐藏要件/消极要件/例外要件
  - 事实槽位（fact_slot）
  - 举证责任方（burden_party）
  - 证明标准（proof_standard）
  - 建议证据类型
- **兜底机制**：如果知识库中 L4 要件为空，LLM 基于法条文本自行拆解

**输出**：`Step4Output`（element_matrix）

**上下文**：原始案件信息 + Step1~Step3 输出

---

#### **Step 5：诉讼主张检索**

**输入**：element_matrix + claim_facts

**处理**：
- 将要件映射到需要证明的事实
- 匹配当事人已主张的事实
- 标注主张状态：asserted（已主张）/ missing（缺失）/ vague（模糊）/ conflicting（矛盾）
- 对缺失/模糊事实生成释明问题

**输出**：`Step5Output`（claim_fact_mapping）

**上下文**：原始案件信息 + Step1~Step4 输出

---

#### **Step 6：争点整理**

**输入**：element_matrix + claim_fact_mapping + defense_basis + evidence_list

**处理**：
- 识别事实争点和法律争点
- 关联要件、诉求、抗辩、证据
- 确定争点优先级（high/medium/low）
- 确定审理顺序
- 控制争点颗粒度（以构成要件为单位）

**输出**：`Step6Output`（issues、review_order）

**上下文**：原始案件信息 + Step1~Step5 输出

---

#### **Step 7：举证质证**

**输入**：issues + element_matrix + evidence_list + cross_examinations

**处理**：
- 为每个要件/争点制定举证计划
- 匹配现有证据
- 识别证明缺口（proof_gap）
- 建议补充证据类型
- 标注真伪不明时的法律后果

**输出**：`Step7Output`（proof_plan）

**上下文**：原始案件信息 + Step1~Step6 输出

---

#### **Step 8：事实认定**

**输入**：proof_plan + evidence_list + cross_examinations

**处理**：
- 对每个要件事实进行认定
- 采信/不采信证据并说明理由
- 标注认定状态：proved（已证明）/ not_proved（未证明）/ unknown（真伪不明）
- 处理真伪不明时的举证责任后果
- 审查自认事实是否涉及国家利益、公共利益等

**输出**：`Step8Output`（fact_findings）

**上下文**：原始案件信息 + Step1~Step7 输出

---

#### **Step 9：要件归入并裁判**

**输入**：fixed_claims + request_basis + defense_basis + element_matrix + fact_findings + issues + 联网搜索法条

**处理**：
1. **联网搜索**（如果启用）：搜索裁判相关法条（top-k=2）
2. 逐项将认定的事实归入要件
3. 审查请求权是否成立
4. 审查抗辩是否成立
5. 适用 L5 法律效果标签
6. 生成裁判结论（supported / partially_supported / rejected / procedural_dismissal）
7. 构建裁判文书框架（诉讼请求/辩称/争议焦点/本院查明/本院认为/判决主文）
8. 执行"八个一致"校验

**输出**：`Step9Output`（subsumption_results）

**上下文**：原始案件信息 + Step1~Step8 输出

**联网搜索**：
- 为每个诉请（最多2个）搜索裁判相关法条
- 查询条件：claim_text + case_cause + legal_domain + claim_type

### 4.4 保底裁判机制详解

#### 评分维度（满分 100 分）

| 维度 | 满分 | 说明 |
|------|------|------|
| claim_clarity | 20 | 诉求清晰度（是否明确、可执行、有具体金额/行为） |
| legal_relation_stability | 15 | 法律关系稳定性（legal_domain、case_cause 是否明确） |
| request_basis_stability | 15 | 请求权基础稳定性（是否找到合适的请求权基础规范） |
| defense_path_completeness | 15 | 抗辩路径完整性（答辩是否充分、是否有抗辩基础） |
| element_fact_coverage | 15 | 要件事实覆盖率（要件对应的事实主张是否完整） |
| evidence_coverage | 15 | 证据覆盖率（待证事实是否有证据支持） |
| fact_finding_reliability | 10 | 事实认定可靠性（已认定事实的比例） |

#### 评分方式

**优先使用 LLM 评分**（`_score_with_llm`）：
- 将 Step1~Step8 的中间产物作为输入
- 调用 LLM 按七维度打分
- 失败时回退到规则法兜底

**规则法兜底评分**（`_score_with_rules`）：
- 基于各步骤输出的结构化数据自动计算
- 逻辑保守，确保评分可靠性

#### 评分等级与分支

| 评分范围 | 等级 | 行为 |
|----------|------|------|
| ≥ 80 | strong | 直接执行 Step 9，输出强裁判 |
| 60-79 | medium | 默认继续 Step 9，附风险提示；允许用户改选弱裁判或部分输出 |
| 40-59 | weak_optional | 必须由用户选择：补充材料/弱裁判/部分输出 |
| < 40 | block | 硬性阻断，仅允许部分输出 |

#### 用户选择选项

| 选项 | 说明 |
|------|------|
| `supplement` | 补充材料，阻断裁判，返回补正清单 |
| `continue_weak_judgment` | 在材料不足情况下继续生成弱裁判（附风险提示） |
| `partial_output_only` | 仅输出要件、争点、证据缺口，不输出裁判倾向 |
| `proceed_with_risk_notes` | 接受风险继续强裁判（medium 等级默认行为） |

#### 硬性拦截条件

满足以下任一条件即触发硬性拦截（`status=blocked`）：
- 案件输入中没有任何诉讼请求（claims 为空）
- 没有提供任何当事人信息（party_info 为空）
- 所有诉讼请求都不可执行（所有 fixed_claims.is_executable = False）

#### 弱裁判生成

当用户选择 `continue_weak_judgment` 时：
1. 调用 LLM 生成弱裁判结论（使用专门的 WEAK_JUDGMENT_SYSTEM prompt）
2. 必须列明：假设前提、缺失输入、未支持要件、证据缺口、各类风险
3. 每个弱裁判结果包含：tentative_judgment_result、confidence、fallback_path
4. 提供升级到强裁判的要求（upgrade_to_strong_judgment_requirements）
5. 失败时回退为 partial_output

## 五、知识库检索机制

### 5.1 五层标签体系（L1-L5）

```
L1: 法源定位与案由关系层
├── source_type              # 法源类型（法律/司法解释/行政法规等）
├── effective_status         # 效力状态（现行有效/已废止等）
├── legal_domain             # 法律领域（合同/侵权/物权等）
├── case_cause_l1~l4         # 案由层级映射
└── special_priority         # 特别优先标记

L2: 九步法位置与规范功能层
├── workflow_steps           # 适用的九步法步骤
└── norm_type                # 规范功能（request_basis/defense_basis/definition等）

L3: 请求/抗辩对象层
├── claim_type               # 请求类型（payment_claim/damages_claim等）
├── defense_type             # 抗辩类型（limitation_defense/setoff_defense等）
├── right_type               # 权利类型
├── liability_type           # 责任类型
└── party_role               # 当事人角色

L4: 构成要件与证明层
├── elements                 # 构成要件列表
│   ├── element_id           # 要件ID
│   ├── element_name         # 要件名称
│   ├── element_logic        # 逻辑关系（AND/OR/NOT）
│   ├── burden_party         # 举证责任方
│   ├── proof_standard       # 证明标准
│   └── suggested_evidence_types  # 建议证据类型
└── ...

L5: 法律效果与裁判输出层
├── legal_effect_tags        # 法律效果标签
├── disposition_type         # 裁判类型
├── effect_if_satisfied      # 要件满足时的效果
├── effect_if_not_satisfied  # 要件不满足时的效果
└── calculation_formula      # 计算公式
```

### 5.2 检索打分策略

**检索流程**：

1. **候选集合构建**：用 `workflow_step` 做第一道硬过滤
2. **效力过滤**：仅保留"现行有效"的规则单元
3. **规范类型过滤**：如必须为 request_basis 或 defense_basis
4. **加权打分**：

| 匹配维度 | 权重 | 说明 |
|----------|------|------|
| workflow_step 命中 | +1 | 基础分 |
| norm_type 命中 | +3/个 | 规范功能匹配 |
| claim_type 命中 | +5/个 | 请求类型匹配 |
| defense_type 命中 | +5/个 | 抗辩类型匹配 |
| case_cause_l4 命中 | +4/个 | 案由四级匹配 |
| case_cause_l3 命中 | +3/个 | 案由三级匹配 |
| case_cause_l2 命中 | +2/个 | 案由二级匹配 |
| case_cause_l1 命中 | +1/个 | 案由一级匹配 |
| legal_domain 命中 | +2/个 | 法律领域匹配 |
| special_priority="特别规则" | +1 | 特别法优先 |
| keyword_hints 命中 | +0.5/个 | 关键词兜底匹配 |

5. **排序返回**：按总分降序，返回 top-k 个规则单元

### 5.3 联网搜索增强

**功能**：使用 DeepSeek V4 Flash 联网搜索最新的中国法律法规，补充本地知识库的不足。

**启用步骤**：
- Step 2：请求权基础检索
- Step 3：抗辩权基础检索
- Step 9：要件归入裁判

**配置参数**（`config/settings.py`）：
```python
ENABLE_WEB_SEARCH = True              # 是否启用联网搜索
WEB_SEARCH_MODEL = "dashscope-deepseek"  # 使用 deepseek-v4-flash
WEB_SEARCH_TOP_K = 2                  # 返回2个最相关法条
```

**搜索流程**：
1. 构建搜索查询（query_text + case_cause + legal_domain + claim_type/defense_type）
2. 调用 LLM 联网搜索（temperature=0.1，max_tokens=8192）
3. 解析返回的法条（law_name、article_no、article_text、effective_status、relevance_reason）
4. 格式化为 prompt 文本，补充到 LLM 输入中

**返回法条结构**：
```json
{
  "law_name": "法律名称",
  "article_no": "条文号",
  "article_text": "法条原文",
  "effective_status": "现行有效/已废止/尚未生效",
  "relevance_reason": "与本案的关联性说明",
  "source_url": "来源URL（如有）"
}
```

**优势**：
- 获取最新法律法规（包括最新司法解释和修正案）
- 补充本地知识库未覆盖的法条
- 提高裁判的时效性和准确性

## 六、LLM 调用机制

### 6.1 配置

- **默认模型**：`qwen3.5-plus`（通过 `dashscope-qwen-plus` ID 引用）
- **可选模型**：
  - `qwen3.6-max-preview`
  - `qwen3.6-plus`
  - `qwen3.6-flash`
  - `deepseek-v4-flash`（用于联网搜索）
- **API 提供商**：DashScope（阿里云，OpenAI 兼容模式）
- **Base URL**：`https://dashscope.aliyuncs.com/compatible-mode/v1`

### 6.2 调用参数

| 参数 | 值 | 说明 |
|------|-----|------|
| temperature | 0.2 | 法律推理偏低温，保证输出稳定性 |
| max_tokens | 12800 | 最大输出 token 数（支持长文本输出） |
| response_format | json_object | 强制 JSON 输出 |
| timeout | 180s | 单次调用超时 |
| max_retries | 3 | 最大重试次数 |
| retry_base_delay | 2s | 重试基础延迟（指数退避） |

### 6.3 容错机制

**JSON 解析容错**：
- 自动去除 markdown 代码块包裹（````json ... ````）
- 提取最外层 JSON 块（第一个 `{` 到最后一个 `}`）
- 修复尾随逗号（`,}` → `}`）
- 支持数组格式（`[...]`）

**调用失败容错**：
- 指数退避重试（2s → 4s → 8s）
- 捕获 RateLimitError、APIError、ValueError
- response_format 不支持时回退为文本模式
- 每步异常记录到 `state.errors`，不阻断后续步骤

## 十、关键设计原则

### 10.1 数据驱动

- 所有法律推理基于输入的案件材料和知识库规则单元
- LLM 不"自由发挥"，严格按 Prompt 模板和 JSON Schema 输出
- 标签值使用预定义枚举，禁止自创新标签

### 10.2 容错优先

- 每步异常不阻断整体流程，记录到 `state.errors` 继续执行
- LLM 调用失败自动重试（指数退避）
- JSON 解析容错（去除代码块、修复尾随逗号等）
- 评分失败回退到规则法兜底
- 弱裁判生成失败回退为 partial_output

### 10.3 渐进式裁判

- 根据输入材料充足度动态调整裁判强度
- 材料充足 → 强裁判（完整裁判文书框架）
- 材料中等 → 中风险裁判（附风险提示）
- 材料不足 → 弱裁判或部分输出
- 材料严重缺失 → 硬性阻断，返回补正清单

### 10.4 可追溯性

- 每步中间产物完整保留，可追溯推理链路
- 记录每步耗时、错误、警告
- 裁判结果关联到具体要件、事实、证据、法条
- 上下文传递机制保证信息完整性

### 10.5 知识增强

- 本地知识库（五层标签体系）提供结构化法律推理基础
- 联网搜索补充最新法律法规
- 两者结合确保法律适用的准确性和时效性

   - 本地检索：从知识库检索买卖合同相关规则单元
   - 联网搜索：搜索最新买卖合同法律法规

   - 上下文：原始案件信息 + Step1 输出

3. **Step 3**：识别被告的同时履行抗辩权
   - 本地检索：抗辩权基础规则单元
   - 联网搜索：同时履行抗辩权相关法条

## 十二、常见问题

### Q1: 法条库文件找不到怎么办？

确保 `articles_annotated.jsonl` 已放置到 `./data/` 目录下，或通过环境变量 `JIUBUFA_KB_PATH` 指定路径。

### Q2: LLM 调用失败怎么办？

检查：
1. API Key 是否正确（`config/settings.py` 或环境变量 `DASHSCOPE_API_KEY`）
2. 网络是否可访问 DashScope API
3. 查看日志中的具体错误信息

### Q3: 如何切换模型？

修改 `config/settings.py` 中的 `DEFAULT_MODEL_ID`，或在 API 请求中传入 `model_name` 参数。

### Q4: 工作流返回 `awaiting_user_choice` 是什么意思？

表示输入材料充足度评分在 40-59 分（weak_optional），需要用户选择后续处理方式：
- `supplement`：补充材料
- `continue_weak_judgment`：继续生成弱裁判
- `partial_output_only`：仅查看要件和争点

前端需渲染 `fallback_gate.available_choices` 让用户选择，然后重新提交带 `fallback_user_choice` 字段的请求。

### Q5: 如何调试某个步骤？

设置日志级别为 DEBUG：
```bash
export JIUBUFA_LOG_LEVEL="DEBUG"
```

查看日志中对应步骤的详细输出，包括 LLM 请求和响应。

### Q6: 联网搜索失败怎么办？

检查：
1. 确认 `ENABLE_WEB_SEARCH = True`
2. 检查 `WEB_SEARCH_MODEL` 是否配置为支持联网的模型（如 deepseek-v4-flash）
3. 查看日志中的错误信息
4. 联网搜索失败不影响主流程，系统会继续使用本地知识库

### Q7: 上下文太长导致 token 过多怎么办？

系统已做优化：
- 事实主张截断为 3000 字符（可在 `context_builder.py` 调整 `max_case_facts_length`）
- 证据清单和庭审笔录使用简化版
- 前面步骤输出只提取关键信息
- 如果仍然过长，可在 `context_builder.py` 中调整截断策略

## 十三、总结

本项目实现了一个完整的 AI 辅助审判推理系统，核心特点：

1. **结构化推理**：严格遵循九步法逻辑，每步产出明确的中间产物
