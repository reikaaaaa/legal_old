# 02 · 材料规范层审查说明

---

## 1. 本层定位

材料规范层是整个系统的**入口关卡**，模拟法院立案庭对起诉材料的初步审查工作。它接收当事人以口语化、碎片化方式提交的案件描述，完成三项核心任务：

- **审核**：判断材料是否满足立案和后续审判分析的最低要求
- **识别**：自动推断案由和法律领域
- **抽取**：从非结构化文本中提取结构化法律要素

本层不调用九步法推理，仅做**材料层面的形式审查 + 信息抽取**。

---

## 2. 输入内容

| 字段 | 类型 | 说明 |
|------|------|------|
| 原始材料文本 | 自由文本 | 当事人的口语化案件描述，无格式要求 |
| 用户身份（可选） | 枚举 | 原告/被告/第三人/代理人/其他 |

**输入示例**（简述）：
> "我是做小生意的，有个客户李某从我这买了混凝土，欠我8万块钱一直不给。当时也没签什么正式合同，就是微信上说的。送了三次货，分别是去年9月20号、9月25号、9月27号。我有微信聊天记录..."

---

## 3. 处理逻辑

### 阶段一：材料审核

**步骤 1 — 识别案件类型**
系统根据用户描述中的关键词和法律关系特征，将案件归入以下类型之一：

> 合同纠纷 / 婚姻家庭 / 侵权纠纷 / 劳动争议 / 民间借贷 / 其他民事案件 / 无法确定

**步骤 2 — 案由专项材料核对（维度一）**
系统对照该案由的法定材料清单，逐项检查：

```
┌────────────────────────────────────────────────────┐
│  以"民间借贷"案由为例，检查项包括：                    │
│  · 原告主体资格证明（身份证/营业执照）               │
│  · 被告身份信息（姓名+住址/工商信息）                │
│  · 借贷合意证明（借条/合同/微信记录）                │
│  · 款项交付证明（转账记录/收条/证人证言）            │
│  · 利息约定证明（如有）                             │
│  · 催收记录（如涉诉讼时效）                         │
└────────────────────────────────────────────────────┘
```

每项标注状态：**已提供 / 部分提供 / 缺失**，并区分"核心材料"（缺失直接影响立案）与"辅助材料"（缺失影响胜诉概率但不影响立案）。

**步骤 3 — 九步法输入需求诊断（维度二）**
逐步骤预判当前材料是否足以支撑后续九步法分析：

> - 第一步：诉请是否明确、充分、无矛盾？
> - 第二步：原告主张的法律关系是否可识别？
> - 第三步：被告回应是"否认"还是"抗辩"？（关键区分）
> - ...
> - 第七步：证据是否落入《民事证据规定》第90条"不能单独定案"情形？
> - ...

**步骤 4 — 综合判定**
`can_proceed = true` 需同时满足四项条件：

| 条件 | 法律依据 |
|------|---------|
| 核心材料提供率 ≥ 80% | — |
| 第一步至第七步无"严重缺失" | — |
| 被告身份信息已提供 | 《民诉法》第122条（有明确的被告） |
| 不处于"仅有当事人陈述而无其他证据"状态 | 《民事证据规定》第90条 |

### 阶段二：材料规范化

将口语化文本抽取为结构化案件数据，输出包含：

- **当事人信息**：姓名、角色（原告/被告）、法律地位（自然人/法人）、与案件关系
- **诉讼请求**：原始表述 + 规范化表述 + 请求类型 + 标的金额
- **事实主张**：逐条拆分，标注时间、行为主体、相对方、事实类型
- **证据清单**：证据名称、提交方、证据类型、证明目的
- **答辩意见**（如有）：答辩方、答辩内容、答辩类型预判

### 核心处理约束

系统在材料规范层设置了**五项硬性约束**：

| 约束 | 说明 |
|------|------|
| **来源标注** | 每个结构化字段均可追溯到原始材料的对应段落 |
| **缺口标注** | 缺失、模糊、矛盾的信息显式标记，不自行填补 |
| **不增不删** | 不在原始材料之外添加当事人未提及的事实 |
| **中立性** | 不预设原告或被告立场，等距处理双方陈述 |
| **可追溯** | 每个抽取元素保留 original_input 字段供回溯核对 |

## 4. 输出内容

### 阶段一输出：MaterialReviewResult

| 字段 | 说明 | 示例值 |
|------|------|--------|
| `case_module` | 识别的案件类型 | "民间借贷" |
| `case_type_check.checklist` | 案由专项材料核对清单 | 6项材料的逐项状态 |
| `case_type_check.core_provided_rate` | 核心材料提供率 | 0.75 (75%) |
| `step_checks[1-9]` | 九步法每步的输入充足度诊断 | 每步含 status + missing_items |
| `overall_status` | 综合评定 | "材料基本完整" |
| `can_proceed` | 是否可进入九步法 | true / false |
| `missing_core_materials` | 缺失的核心材料清单 | ["被告身份信息"] |
| `upload_instructions` | 补正指引（通俗语言） | "请通过国家企业信用信息公示系统下载被告工商查询单..." |

### 阶段二输出：NormalizedCaseInput

完全对齐九步法入口所需的 CaseInput 数据模型，包含：party_info, claims, claim_facts, defense_opinions, evidence_list, evidence_meta, cross_examinations, court_records, legal_arguments, procedural_info 等。

## 5. 需要律师重点审查的问题

### 5.1 材料充足性判断

| # | 审查问题 | 审查要点 |
|---|---------|---------|
| 1 | 系统判断"材料不足"的标准是否合理？ | 请对照《民诉法》第122条立案条件，确认系统对"必须材料"的定义是否准确 |
| 2 | 哪些材料属于**起诉必须材料**？ | 立案审查 vs 实体审理的材料标准不同，系统是否混淆？ |
| 3 | 哪些材料缺失应提示补充，而非直接中断流程？ | 材料不足时应"补正"还是"驳回"？请判断系统的 can_proceed 阈值(80%)是否合理 |
| 4 | "仅有当事人陈述"的判断是否过于严格？ | 部分案件中当事人陈述+间接证据也具有一定证明力 |

### 5.2 案由识别

| # | 审查问题 | 审查要点 |
|---|---------|---------|
| 1 | 系统根据用户描述识别案由，是否符合实务分类？ | 请用附带的案例1检验案由识别是否正确 |
| 2 | 是否需要区分更细的案由？ | 例如"合同纠纷"下是否应区分"买卖合同纠纷""服务合同纠纷"等四级案由 |
| 3 | **案由识别错误是否会导致后续九步法路径全部偏移？** | 这是关键风险点，请重点评估 |

### 5.3 诉讼请求拆分

| # | 审查问题 | 审查要点 |
|---|---------|---------|
| 1 | 系统对原始诉请的拆分是否合理？ | 是否准确拆分为独立请求项？ |
| 2 | claim_text_normalized 是否保持了原始请求的法律含义？ | 规范化不应改变诉请实质 |
| 3 | claim_type 归类（金钱给付/行为给付/确认请求等）是否准确？ | 归类错误可能影响后续请求权基础匹配 |
| 4 | 金额提取是否准确？ | 尤其注意部分请求、选择性请求的情况 |
| 5 | 系统是否把**事实主张误写成诉讼请求**？ | 例如"他欠我钱"是事实主张，"请求判令归还"是诉讼请求 |

---

## 6. 系统边界

| 系统不做 | 原因 |
|---------|------|
| 判断证据真实性 | 仅标注"待核实"，不做终局性判断 |
| 判断当事人主体是否适格 | 仅提示需提供主体资格证明，不做实体审查 |
| 替代律师撰写起诉状 | 仅提取和整理，不生成法律文书 |
| 判断是否超过诉讼时效 | 仅提示需提供催收记录，不主动适用时效规则 |
| 自行补充缺失的当事人信息 | 只提示缺失，不编造 |

## 7. 案例对照审查指引

请打开 `demo_cases/案例1_原始材料.txt`（软件著作权侵权案的原始输入），对照 `demo_cases/案例1_材料规范层输出.json`，逐项检查：

1. 案由识别是否正确？（应识别为"侵害计算机软件著作权纠纷"）
2. 证据清单是否完整抽取了所有提及的证据？
3. 诉讼请求拆分是否正确？（停止侵害 + 赔偿2712827元）
4. 缺失材料提示是否合理？（被告工商登记信息确为缺失）
5. 事实拆分是否保留了妨害证据保全这一关键情节？
