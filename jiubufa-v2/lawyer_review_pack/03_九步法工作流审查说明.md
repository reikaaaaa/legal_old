# 03 · 九步法工作流审查说明

---

## 本层定位

九步法推理引擎是系统的**核心分析层**，严格按照"要件审判九步法"方法论串行执行 Step1 至 Step9。每一步模拟法官/律师在审理案件时的真实思维过程，从前端诉请固定到最终裁判归入，形成完整的法律推理链条。

> **审查关键**：每一步的法律逻辑是否符合真实审判流程？步骤之间的数据流转是否正确？有无跳跃、遗漏或逻辑倒置？

---

## Step 1：固定权利请求与请求类型

### 输入
- 材料规范层输出的 `claims` 字段（原始诉请 + 当事人身份）

### 处理逻辑
1. 从当事人陈述中识别每一项独立的诉讼请求
2. 将口语化表述规范化为法律语言（保留原始含义）
3. 判断每项请求是否**明确**（诉求具体）、**可执行**（有明确的义务人和给付内容）
4. 将请求归类到 L3 类型：确认请求/金钱给付/行为给付/形成请求等
5. 识别请求权竞合、聚合、备位关系
6. 推断案由和法律领域

### 输出
| 字段 | 说明 |
|------|------|
| `fixed_claims[].claim_id` | 请求编号（C1, C2...） |
| `fixed_claims[].claim_text_normalized` | 规范化后的请求表述 |
| `fixed_claims[].claim_type` | 请求类型（injunction_claim, damages_claim 等） |
| `fixed_claims[].is_clear` | 是否明确 |
| `fixed_claims[].is_executable` | 是否可执行 |
| `fixed_claims[].competition_note` | 请求权竞合/聚合说明 |
| `case_cause_inferred` | 推断的案由 |

### 律师审查问题

| # | 问题 | 审查要点 |
|---|------|---------|
| 1 | 诉请拆分是否准确？有无遗漏或多余？ | 对照原始材料，诉请是否全部被识别 |
| 2 | 规范化后的请求是否改变了原始含义？ | 警惕"过度规范化"导致诉请变形 |
| 3 | 请求权竞合/聚合的识别是否合理？ | 例如同时主张违约金和赔偿金是否构成竞合 |
| 4 | 案由推断是否符合《民事案件案由规定》？ | 合同纠纷/侵权纠纷的区分是否正确 |

## Step 2：确定权利请求基础规范

### 输入
- Step1 输出的 `fixed_claims` + `case_cause_inferred`

### 处理逻辑
1. 在 4,559 条法条知识库中，按**五层标签**（L1法律部门→L2法律领域→L3制度→L4法条群→L5规则单元）加权检索
2. 为每项请求匹配最匹配的请求权基础法条（如《民法典》第577条→违约责任一般条款）
3. 优先适用：具体规则优于一般规则，特别法优于一般法
4. 联网搜索补充不在知识库中的最新法条

### 输出
| 字段 | 说明 |
|------|------|
| `request_basis_candidates[].rule_unit_id` | 匹配的法条ID |
| `request_basis_candidates[].law_name` | 法规名称 |
| `request_basis_candidates[].article_no` | 条文编号 |
| `request_basis_candidates[].rule_unit_text` | 条文原文 |
| `request_basis_candidates[].relevance_score` | 匹配度评分 |
| `request_basis_candidates[].selection_reason` | 选择理由 |

### 律师审查问题

| # | 问题 | 审查要点 |
|---|------|---------|
| 1 | 每条诉请匹配的法条是否准确？ | 请求权基础是否正确？有无张冠李戴？ |
| 2 | 是否存在**漏引**重要法条？ | 例如遗漏特别法规定 |
| 3 | "特别法优于一般法"的判断是否正确？ | 例如消费者权益纠纷中是否优先适用《消法》 |
| 4 | 匹配度评分高的法条是否真的最合适？ | 评分机制可能偏好字面匹配而非实质匹配 |

## Step 3：确定抗辩权基础规范

### 输入
- Step1/Step2 输出 + 被告答辩意见

### 处理逻辑
系统对被告的每一条答辩做**五分类**：

| 类型 | 定义 | 举例 | 对举证责任的影响 |
|------|------|------|----------------|
| **承认** | 对原告主张事实表示认可 | "对，欠这个钱" | 免除原告对该事实的举证责任 |
| **否认** | 对原告主张事实表示否定 | "我没借过钱" | 原告继续承担举证责任 |
| **抗辩** | 主张新事实以对抗原告诉求 | "借过但已经还了" | 被告对抗辩事实承担举证责任 |
| **抗辩权** | 主张有阻却请求权的权利 | "已过诉讼时效" | 被告对时效届满承担举证责任 |
| **程序性异议** | 对程序合法性提出异议 | "法院没管辖权" | 被告对程序瑕疵承担举证责任 |

对于真正的"抗辩"和"抗辩权"，系统进一步检索抗辩权基础规范。

### 输出
| 字段 | 说明 |
|------|------|
| `defense_basis_candidates[].response_type` | 承认/否认/抗辩/抗辩权/程序性异议 |
| `defense_basis_candidates[].defense_type` | 抗辩的具体类型（时效抗辩/履行完毕抗辩等） |
| `defense_basis_candidates[].rule_unit_ref` | 抗辩权基础法条（如有） |

### 律师审查问题（**重点**）

| # | 问题 | 审查要点 |
|---|------|---------|
| 1 | **否认 vs 抗辩的区分是否准确？** | 这是最体现专业性的地方，区分错误会直接导致举证责任分配错误 |
| 2 | 是否需要增加"部分承认、部分否认"的情况？ | 实务中常见：承认借款但否认金额 |
| 3 | 是否需要区分"事实否认"与"法律否认"？ | 例如"我没签合同"vs"合同无效" |
| 4 | 系统对"无答辩"情况的处理是否合理？ | 被告未答辩时系统如何处理举证责任？ |

## Step 4：分析基础规范构成要件

### 输入
- Step2/Step3 选定的法条

### 处理逻辑
将每条选定法条拆解为**构成要件要素**，区分：

- **积极要件**：需原告证明的要素
- **消极要件**：需被告证明的要素（如免责事由）
- **隐含要件**：法条未明示但逻辑必需（如合同关系成立）
- **例外要件**：法条的但书或除外情形

每个要素标注：`element_type`（主体/法律关系/行为/违约/过错/损害/因果等）、`element_logic`（AND/OR/NOT）、`burden_party`、`proof_standard`

### 输出
| 字段 | 说明 |
|------|------|
| `element_matrix[].element_id` | 要件编号 |
| `element_matrix[].element_name` | 要件名称 |
| `element_matrix[].element_type` | 要件类型 |
| `element_matrix[].burden_party` | 举证责任方 |
| `element_matrix[].proof_standard` | 证明标准 |

### 律师审查问题

| # | 问题 | 审查要点 |
|---|------|---------|
| 1 | 要件拆解是否完整？有无遗漏隐含要件？ | 如合同纠纷中"合同关系成立"是隐含前提 |
| 2 | 举证责任分配是否正确？ | 是否符合"谁主张谁举证"？ |
| 3 | 证明标准是否合理？ | 一般民事"高度盖然性"，特殊类型可能有不同标准 |

---

## Step 5：诉讼主张检索

### 输入
- Step4 要件矩阵 + 当事人事实主张

### 处理逻辑
将当事人的每项事实主张，**映射**到其应该证明的构成要件上。标注映射状态：

| 状态 | 含义 |
|------|------|
| `asserted` | 已主张且明确 |
| `missing` | 完全没主张 |
| `vague` | 主张但不特定化 |
| `conflicting` | 主张前后矛盾 |

### 输出
`claim_fact_mapping[]`，每个要件对应的事实ID和状态，缺失/模糊/矛盾时附带 `clarification_question` 和 `risk_note`。

### 律师审查问题

| # | 问题 |
|---|------|
| 1 | 事实到要件的映射是否一一对应？有无跨要件的事实？ |
| 2 | `missing` 的事实是否确实缺失？还是被误判为缺失？ |
| 3 | `vague` 的阈值是否合理？ |

---

## Step 6：争点整理

### 输入
- 前5步累积的所有数据

### 处理逻辑
识别并分类所有**有实质争议**的问题：

- **事实争点**（fact_issue）：对事实是否存在有争议
- **法律争点**（legal_issue）：对法律适用有争议

每个争点标注 `burden_party`（举证责任方）、`priority`（high/medium/low）、关联证据ID。输出 `review_order`（庭审审查顺序建议）。

### 律师审查问题

| # | 问题 |
|---|------|
| 1 | 争点颗粒度是否合理？有没有把"诉请是否成立"当成唯一大争点？ |
| 2 | 每个"事实争点"是否真正存在双方争议（而非单方主张）？ |
| 3 | 审查顺序是否合理（先审主体、再审关系、后审违约损害）？ |
| 4 | 是否遗漏了关键争点（如妨害保全是否可以推定侵权）？ |

## Step 9：要件归入并作出裁判

### 输入
- 前8步全部输出 + 联网搜索补充法条

### 处理逻辑
严格按照**三段论**逐项归入：

```
大前提（法条）：《著作权法》第53条 → 侵权人应承担停止侵害、赔偿损失等民事责任
小前提（事实）：F1 原告系著作权人 + F2 被告安装侵权软件 + F3 妨害保全推定
结论（裁判）：支持原告停止侵害请求
```

每个构成要件标注 `finding_status`（proved/disproved/unknown），逐项推理后得出 `judgment_result`（supported/partially_supported/rejected/procedural_dismissal），输出 `reasoning_summary`（"本院认为..."说理文本）。

### 律师审查问题

| # | 问题 | 审查要点 |
|---|------|---------|
| 1 | 三段论归入的逻辑是否严密？ | 每个要件→事实→结论的链条是否完整 |
| 2 | 是否存在**漏判**（有诉请无结论）？ | 一致性校验应标记此项 |
| 3 | 是否存在**超判**（无诉请有结论）？ | 不告不理原则 |
| 4 | 说理文本是否使用了正式裁判语言？ | 是否适合作为裁判文书参考？ |
| 5 | 引用的法条是否实际支持该结论？ | 法条与结论之间的逻辑关系 |

## 保底裁判门

在 Step8 之后、Step9 之前，系统计算**7维度输入完整度评分**（0-100分）：

| 维度 | 分值 | 含义 |
|------|------|------|
| claim_clarity | 0-20 | 诉讼请求是否具体、明确、可执行 |
| legal_relation_stability | 0-15 | 案由和法律关系是否可识别 |
| request_basis_stability | 0-15 | 是否找到稳定的请求权基础 |
| defense_path_completeness | 0-10 | 抗辩/否认/程序异议是否识别完整 |
| element_fact_coverage | 0-15 | 构成要件是否有事实主张覆盖 |
| evidence_coverage | 0-15 | 证据是否覆盖核心待证事实 |
| fact_finding_reliability | 0-10 | 是否能可靠作出证明状态判断 |

根据评分进入四个通道：

| 评分 | 通道 | 行为 |
|------|------|------|
| ≥80 | 强裁判 | 完整 Step9 裁判输出 |
| 60-79 | 中风险裁判 | 正常输出 + 风险提示 |
| 40-59 | 用户选择门 | 暂停，提供补充/弱裁判/部分输出三个选项 |
| <40 | 硬阻断 | 仅输出缺口清单，不输出裁判 |

### 律师审查问题

| # | 问题 |
|---|------|
| 1 | 7个维度的分值分配是否合理？ |
| 2 | 四级阈值设定（80/60/40）是否符合法律实务？ |
| 3 | 中风险时"附风险提示继续"是否安全？ |
