## 一、系统资源画像

| 维度 | 数值 | 说明 |
|------|------|------|
| 后端框架 | FastAPI + Uvicorn | Python 3.12，同步模式 |

## 二、推荐方案：Docker Compose + 云 ECS

### 2.1 为什么选这个

| 对比方案 | 是否适合 | 原因 |
|---------|---------|------|
| **纯 VPS 裸部署** | ❌ | Python 依赖管理混乱，缺少环境隔离 |
| **Serverless (FC/Lambda)** | ❌ | 单次推理 8-12 分钟，远超 30s 超时限制 |
| **K8s 集群** | ❌（过度设计） | 初期 < 10 并发，单机完全够用 |
| **Docker Compose 单机** | ✅ **推荐** | 环境一致、部署简单、水平扩展只需加实例 |
| **PaaS（Railway/Render）** | ⚠️ 可选 | 方便但跨境延迟大，国内建议用阿里云 |

### 2.2 推荐云服务商：阿里云（与 DashScope 同区域）

| 资源 | 规格建议 | 月费估算 |
|------|---------|---------|
| ECS 实例 | 4 核 8G，系统盘 40G + 数据盘 100G | ~¥300-500/月 |
| 公网 IP + 带宽 | 按流量计费或 5Mbps 固定 | ~¥100-200/月 |
| 对象存储 OSS（可选） | 标准存储，存放用户上传文件 | ~¥10-50/月 |
| **合计** | | **~¥400-800/月** |

> 为什么不选腾讯云/华为云：DashScope 是阿里云产品，ECS 与 DashScope API 在同一可用区可降低网络延迟 30-50ms/次，9 步叠加后节省 2-5 秒。

## 三、Docker 化组件拆分

### 3.1 容器规划

```
┌─────────────────────────────────────────────────────┐
│                   Nginx (端口 80/443)                 │
│   · 前端静态文件服务                                   │
│   · API 反向代理 (/api/* → backend:8000)              │
│   · HTTPS 终止 (Let's Encrypt / 阿里云 SSL)           │
│   · 客户端限速 + IP 白名单（可选）                      │
└────────────┬────────────────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────────────────┐
│              Backend (FastAPI + Uvicorn)              │
│   · 端口 8000（内部，不暴露公网）                       │
│   · 1 worker（九步法同步串行，多 worker 暂无必要）       │
│   · 健康检查端点 /api/health                          │
│   · 注意：同步模式下 uvicorn 需配置 timeout_keep_alive  │
│     至少 900s（覆盖最慢的一轮 12 分钟）                  │
└────────────┬─────────────────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────────────────┐
│              Redis (可选，Phase 2)                    │
│   · 异步任务队列 Broker（Celery/ARQ）                  │
│   · 结果缓存（相同案由的法条检索结果）                   │
│   · 会话状态存储（多轮补充材料场景）                    │
└─────────────────────────────────────────────────────┘
```

### 3.2 目录挂载

| 宿主机路径 | 容器内路径 | 用途 | 持久化 |
|-----------|-----------|------|--------|
| `./data/` | `/app/data/` | 知识库 JSONL（只读） | 是 |
| `./traces/` | `/app/debug_viewer/traces/` | 调试 Trace | 否（可清理） |
| `./logs/` | `/app/logs/` | 应用日志 | 是 |
| `./uploads/` | `/app/uploads/` | 用户上传材料（如有） | 是 |

### 3.3 Dockerfile 骨架（方向）

- **Base Image**：`python:3.12-slim`（不用 alpine，DashScope SDK 依赖编译环境）
- **依赖安装**：`requirements.txt`（openai / fastapi / uvicorn / dashscope / pydantic）
- **启动命令**：`uvicorn api:app --host 0.0.0.0 --port 8000 --timeout-keep-alive 900`
- **健康检查**：`curl -f http://localhost:8000/api/health`

## 四、关键问题：长耗时请求的处理

### 4.1 问题

单次 `/api/workflow/run` 耗时 8-12 分钟。如果前端直接等待 HTTP 响应，浏览器/反向代理可能在 60-120s 超时断开。

### 4.2 方案：异步任务模式

```
前端 POST /api/workflow/run
        │
        ▼
    Backend 创建任务 → 返回 task_id（立即，200ms）
        │
        ▼
    Celery Worker 异步执行九步法（8-12 min）
        │
        ▼
    结果写入 Redis（task_id → result JSON）
        │
        ▼
    前端轮询 GET /api/workflow/status/{task_id}
        │   ├── status: "running", progress: 3/9, current_step: "争点整理"
        │   └── status: "completed", result: {...}
        │
        ▼
    前端拿到结果，渲染 Stage 3
```

### 4.3 前端适配要点

- Stage 2 提交后不阻塞 UI，显示进度条（轮询 `/api/workflow/status/{task_id}`）
- 每个步骤完成后推送进度增量（Step 1/9...5/9...9/9）
- Demo 模式保持不变（无需后端，零延时的完整演示）

### 4.4 为什么不直接用 WebSocket

- 轮询足够：8-12 分钟/次，每 3-5 秒轮询一次 = 150-240 次请求，对服务器无压力
- WebSocket 增加前后端复杂度，且需要额外处理断线重连、状态同步

## 五、网络与安全

### 5.1 网络拓扑

```
互联网
  │
  ▼
Nginx（80/443）── 仅暴露此端口
  │
  ├── /api/* ──▶ Backend:8000
  │
  └── /* ──▶ 静态文件（app.html + debug viewer）
```

- Backend 和 Redis 不暴露公网端口
- Nginx 负责 HTTPS、限速、请求体大小限制

### 5.2 安全清单

| 项 | 措施 |
|----|------|
| HTTPS | Let's Encrypt 自动续签 或 阿里云免费 SSL 证书 |
| API Key 保护 | DashScope Key 通过环境变量注入，不入 Docker 镜像 |
| 请求体限制 | Nginx `client_max_body_size 10m`（材料文本一般不超过 1MB） |
| 速率限制 | Nginx `limit_req` 限制单 IP 5 次/分钟（防止恶意刷 API） |
| 访问控制 | 初期用 IP 白名单 + Basic Auth，后续接 OAuth2/OIDC |
| CORS | Backend 已配置 `allow_origins=["*"]`，生产环境应收敛为具体域名 |

## 六、多阶段部署路线图

### Phase 1：最小可用部署（1-2 天）

```
目标：能用浏览器访问，能跑完一轮九步法

├── 购买阿里云 ECS（4核8G，CentOS 7.9 或 Ubuntu 22.04）
├── 安装 Docker + Docker Compose
├── 编写 docker-compose.yml（Nginx + Backend 两个服务）
├── 上传知识库 data/articles_annotated.jsonl
├── 配置 .env（DashScope Key）
├── docker compose up -d
└── 验证：浏览器访问 http://<公网IP>/app → Demo 模式 → 真实分析
```

### Phase 2：异步化 + 进度推送（2-3 天）

```
目标：用户提交后不卡在页面等 10 分钟

├── docker-compose 增加 Redis 服务
├── Backend 引入 Celery/ARQ 异步任务队列
├── 新增端点：POST /api/workflow/run → 返回 task_id
├── 新增端点：GET /api/workflow/status/{task_id} → 返回进度
├── 前端适配轮询 + 进度条
└── 验证：提交案例 → 显示进度 1/9→5/9→9/9 → 结果展示
```

### Phase 3：生产加固（2-3 天）

```
目标：能放心给别人用

├── HTTPS + 域名绑定
├── 环境变量管理（DashScope Key / 数据库密码不入仓库）
├── 日志轮转（logrotate，防止磁盘写满）
├── 健康监控（Uptime Kuma 或阿里云云监控）
├── 备份脚本（知识库 + 环境变量，定时备份到 OSS）
├── 启动/重启脚本 + systemd 服务文件
└── 压力测试（模拟 10 并发，确认不出 OOM、不会雪崩）
```

### Phase 4：扩展优化（按需）

```
├── 水平扩展：docker compose up --scale backend=3 + Nginx upstream 负载均衡
├── 引入 PostgreSQL：替代 JSONL 文件存储知识库，支持多用户/多租户
├── GPU 实例：引入 Mist v2 图像对抗保护（需 NVIDIA GPU，费用显著提升）
├── CDN：前端静态文件推到阿里云 CDN / OSS，加速全国访问
└── OAuth2：接入律所统一身份认证系统
```

## 八、风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| DashScope API 不可用 | 系统完全不可用 | 接入多模型备用（DeepSeek API 做 fallback） |
| 单次 10 分钟耗时用户不耐受 | 用户体验差 | Phase 2 异步化 + 进度推送；后续优化到 4-5 分钟 |
| 知识库法条过时 | 引用废止法条 | 定期更新 JSONL + 联网搜索兜底 |
| 并发升高后 OOM | 服务崩溃 | 限制并发 worker 数 + 内存监控告警 |
| Docker 容器逃逸 | 安全隐患 | 非 root 运行容器 + 只读挂载知识库目录 |

## 九、技术选型速查表

| 层面 | 选型 | 备选 |
|------|------|------|
| 容器化 | Docker + Docker Compose | Podman |
| Web Server | Nginx（官方镜像） | Caddy（自动 HTTPS） |
| 反向代理 | Nginx `proxy_pass` | Traefik |
| Python 运行时 | Python 3.12-slim | 3.11-slim（更保守） |
| 异步队列 | Celery + Redis | ARQ（更轻量，纯 Python async） |
| 缓存 | Redis | 无（Phase 1 不需要） |
| 数据库（将来） | PostgreSQL 15 | SQLite（单用户场景够用） |
| 日志 | stdout + Docker logs + logrotate | Loki + Grafana |
| 监控 | Uptime Kuma（自建） | 阿里云云监控 |
| CI/CD | GitHub Actions → rsync 部署 | 手动 `docker compose up -d --build` |
| 域名 DNS | 阿里云 DNS 解析 | Cloudflare（需备案前置条件） |
| SSL | Let's Encrypt + certbot | 阿里云免费 SSL 证书 |
| 对象存储 | 阿里云 OSS | MinIO（自建 S3 兼容存储） |
